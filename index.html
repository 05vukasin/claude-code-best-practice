Section 1

Project Structure

A well-organized .claude/ directory is the foundation of effective Claude Code usage. Here's the recommended layout:

your-project/
├── CLAUDE.md                        # Project instructions
├── .claude/
│   ├── settings.json                # Project settings (commit)
│   ├── settings.local.json          # Local overrides (gitignored)
│   ├── .mcp.json                    # MCP server config
│   ├── skills/                      # Custom slash commands
│   │   ├── deploy/SKILL.md
│   │   └── review-pr/SKILL.md
│   ├── agents/                      # Custom subagents
│   │   ├── code-reviewer/code-reviewer.md
│   │   └── debugger/debugger.md
│   ├── rules/                       # Modular instructions
│   │   ├── code-style.md
│   │   ├── testing.md
│   │   └── security.md
│   ├── hooks/                       # Hook scripts
│   │   └── protect-files.sh
│   └── agent-memory/                # Subagent memory
├── src/
├── tests/
└── .gitignore

User-Level Configuration

Global settings that apply to all your projects live in ~/.claude/:

~/.claude/
├── CLAUDE.md              # Personal instructions (all projects)
├── settings.json          # Global user settings
├── skills/                # Reusable skills across projects
├── agents/                # Reusable subagents across projects
├── rules/                 # Personal rules for all projects
└── projects/
    └── <project-name>/
        └── memory/        # Auto memory (machine-local)
            └── MEMORY.md
Key Principle: Project-level config (.claude/) is shared with your team via git. User-level config (~/.claude/) is personal and stays on your machine.
Section 2

CLAUDE.md — Project Instructions

CLAUDE.md is loaded at the start of every session. It provides persistent instructions that guide Claude's behavior consistently across conversations.

Placement Hierarchy

LocationScopeShared?
~/.claude/CLAUDE.mdAll your projectsNo
./CLAUDE.md or ./.claude/CLAUDE.mdProject rootYes (via git)
./src/CLAUDE.mdSubdirectoryLoaded on-demand

What to Include vs. Exclude

INCLUDE
  • Build, test, deploy commands
  • Code style rules differing from defaults
  • Repository conventions
  • Architecture decisions
  • Common gotchas
  • Required environment variables
EXCLUDE
  • Standard conventions Claude knows
  • Info derivable from reading code
  • Long tutorials or explanations
  • Frequently changing data
  • Detailed API docs (link instead)
  • Self-evident practices

Example CLAUDE.md

# MyApp

## Build & Test
- Install: `npm install`
- Build:   `npm run build`
- Test:    `npm test`
- Lint:    `npm run lint`

## Code Style
- TypeScript for all new code
- 2-space indentation, semicolons
- Use `import`/`export`, not `require`

## Git Workflow
- Branch from `main`: feature/desc or fix/issue-123
- Run tests before committing
- Create PR for review before merging

## Architecture
- API handlers: src/api/handlers/
- Business logic: src/logic/
- Tests colocated: src/module.test.ts

Importing External Files

Use @path syntax to reference other files, keeping CLAUDE.md concise:

See @README.md for setup instructions.
Git workflow: @docs/git-instructions.md
Personal prefs: @~/.claude/my-project-instructions.md
Best Practice: Keep CLAUDE.md under 200 lines. Longer files consume more context and reduce adherence. Split into .claude/rules/ when it grows.
Section 3

Settings Configuration

Settings control permissions, hooks, environment variables, and behavior. They follow a priority hierarchy:

1. CLI Flags (Highest)
--model, --permissions, etc.
2. Local Settings
.claude/settings.local.json — gitignored
3. Project Settings
.claude/settings.json — committed
4. User Settings (Lowest)
~/.claude/settings.json — global

Project Settings Example

{
  "permissions": {
    "allow": [
      "Bash(npm *)",
      "Bash(git *)",
      "Edit(src/**/*.ts)"
    ],
    "deny": [
      "Bash(rm -rf)",
      "Write(.env)",
      "Write(package-lock.json)"
    ]
  },
  "env": {
    "NODE_ENV": "development"
  }
}

User Settings Example

{
  "model": "opus",
  "permissions": {
    "allow": [
      "Bash(git *)",
      "Bash(npm *)"
    ]
  },
  "env": {
    "GITHUB_TOKEN": "$GITHUB_TOKEN"
  }
}
Tip: Use settings.local.json for machine-specific overrides you don't want to commit (API keys, personal preferences).
Section 4

Custom Skills (Slash Commands)

Skills are markdown files that create custom /commands. They extend Claude with domain-specific workflows.

How to Create a Skill

1
Create the directory
mkdir -p .claude/skills/my-skill for project or ~/.claude/skills/my-skill for personal
2
Write SKILL.md
Add YAML frontmatter + markdown instructions
3
Test it
Invoke with /my-skill or /my-skill <arguments>

SKILL.md Format

---
name: fix-issue
description: Fix a GitHub issue by number
argument-hint: [issue-number]
user-invocable: true
allowed-tools: Read, Grep, Glob, Bash, Edit
---

# Fix GitHub Issue

1. Fetch issue details: `gh issue view $ARGUMENTS`
2. Analyze the problem
3. Find relevant code
4. Implement the fix
5. Write tests
6. Create a commit

Frontmatter Reference

FieldDescription
nameSlash command name (lowercase, hyphens)
descriptionWhat it does — Claude uses this for auto-invocation
argument-hintHint shown in autocomplete
user-invocabletrue/false — show in / menu
disable-model-invocationtrue — only manual invocation
allowed-toolsComma-separated tool whitelist
modelsonnet, opus, haiku, or inherit
contextfork — run in isolated subagent
agentSubagent type when context: fork
pathsGlob patterns — only auto-load for matching files

Using Arguments

---
name: migrate-component
description: Migrate a component between frameworks
---

Migrate $0 from $1 to $2.

Invoke: /migrate-component SearchBar React Vue

$ARGUMENTS
All arguments as a single string
$0, $1, $2
Positional arguments (0-indexed)

Dynamic Context & Subagent Execution

---
name: pr-summary
description: Summarize the current pull request
context: fork
agent: Explore
---

## PR Data
- Diff: !`gh pr diff`
- Comments: !`gh pr view --comments`

Summarize this PR, highlighting key changes and risks.

Shell commands in !`command` run before Claude sees the skill. The context: fork runs it in an isolated subagent.

Skill Directory Structure 
my-skill/
├── SKILL.md              # Main instructions (required)
├── template.md           # Optional: output template
├── examples/
│   └── sample.md         # Optional: example outputs
└── scripts/
    └── helper.sh         # Optional: scripts Claude can run

Subagents

Subagents are specialized AI assistants that run in isolated contexts with their own system prompt, tools, and model. Claude delegates tasks to them automatically.

Built-in Subagents

Explore
Read-only. Fast codebase search and analysis without modifications.
Plan
Read-only. Research and plan before implementation.
general-purpose
All tools. Complex multi-step tasks requiring full access.

Creating a Custom Subagent

Create .claude/agents/code-reviewer/code-reviewer.md:

---
name: code-reviewer
description: Expert code review specialist
tools: Read, Grep, Glob, Bash
model: sonnet
---

You are a senior code reviewer. When invoked:

1. Run `git diff` to see recent changes
2. Review for:
   - Code clarity and readability
   - Security vulnerabilities
   - Performance issues
   - Test coverage
3. Provide feedback organized by severity.

nameUnique identifier (lowercase, hyphens)
descriptionWhen Claude should delegate to this agent
toolsAllowed tools (allowlist)
disallowedToolsTools to deny (denylist)
modelsonnet, opus, haiku, or inherit
permissionModedefault, acceptEdits, dontAsk, bypassPermissions
maxTurnsMax agentic turns before stopping
memoryuser, project, or local
backgroundtrue — always run as background task
isolationworktree — run in isolated git worktree

Section 6

Hooks

Hooks are shell commands that execute at specific lifecycle points. Unlike CLAUDE.md instructions which are advisory, hooks provide deterministic control — they always execute.

Hook Events

EventWhen it firesCan Block?
SessionStartSession begins or resumesNo
UserPromptSubmitBefore processing your promptNo
PreToolUseBefore a tool executesYes (exit 2)
PostToolUseAfter a tool succeedsNo
PostToolUseFailureAfter a tool failsNo
NotificationClaude needs your attentionNo
StopClaude finishes respondingNo
SubagentStartSubagent startsNo
SubagentStopSubagent finishesNo

Configuration

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "./.claude/hooks/protect-files.sh"
          }
        ]
      }
    ]
  }
}

Exit Codes

Exit 0
Action proceeds. Stdout injected as context.
Exit 2
Action blocked. Stderr shown as feedback.
Other
Action proceeds. Stderr logged only.

Example: Protect Files

#!/bin/bash
# .claude/hooks/protect-files.sh
INPUT=$(cat)
FILE=$(echo "$INPUT" | jq -r '.tool_input.file_path // empty')

PROTECTED=(".env" "package-lock.json")
for pattern in "${PROTECTED[@]}"; do
  if [[ "$FILE" == *"$pattern"* ]]; then
    echo "Blocked: $FILE is protected" >&2
    exit 2
  fi
done
exit 0

Example: Auto-Format After Edits

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "jq -r '.tool_input.file_path' | xargs npx prettier --write"
          }
        ]
      }
    ]
  }
}

Hook Types

command
Run a shell command
prompt
Single LLM call (no tools)
agent
Full subagent execution
http
POST to HTTP endpoint
Section 7

MCP Servers

The Model Context Protocol connects Claude to external tools and services — databases, APIs, GitHub, Slack, and more.

Configuration

Place in .claude/.mcp.json (project) or ~/.claude/.mcp.json (user):

{
  "mcpServers": {
    "github": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"]
    },
    "postgres": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres"],
      "env": {
        "DATABASE_URL": "$DATABASE_URL"
      }
    },
    "slack": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-slack"],
      "env": {
        "SLACK_BOT_TOKEN": "$SLACK_BOT_TOKEN"
      }
    }
  }
}

Server Types

stdio
Local process via stdin/stdout — CLI tools, npm packages
http
HTTP POST — remote servers, cloud services
sse
Server-Sent Events — real-time data streams
ws
WebSocket — persistent bidirectional connections
Scope: Project MCP (.claude/.mcp.json) is shared with the team. User MCP (~/.claude/.mcp.json) applies to all your projects. You can also scope MCP servers to specific subagents via their frontmatter.
Section 8

Rules (Modular Instructions)

Rules let you split CLAUDE.md into topic-specific files. They live in .claude/rules/ and can be path-specific.

.claude/rules/
├── code-style.md        # Always loaded
├── testing.md            # Always loaded
├── security.md           # Always loaded
└── api/
    └── endpoint-design.md # Path-specific

Path-Specific Rules

Add paths frontmatter to only load rules when Claude works with matching files:

---
paths:
  - "src/api/**/*.ts"
  - "api/**/*.{ts,tsx}"
---

# API Development Rules

- All endpoints must include input validation
- Use standard error format from src/api/errors.ts
- Include OpenAPI documentation comments
- Return consistent pagination for list endpoints
When to use rules vs. CLAUDE.md: Use CLAUDE.md for universal project instructions. Use rules for domain-specific instructions that only apply to certain areas of the codebase.

Memory System

Claude's auto memory accumulates knowledge across sessions — build commands, patterns, debugging insights, and workflow habits.

Storage Location

~/.claude/projects/<project>/memory/
├── MEMORY.md            # Index (first 200 lines loaded at startup)
├── user_role.md         # User information
├── feedback_testing.md  # User feedback
└── project_goals.md     # Project context

Memory Types

user
User's role, preferences, knowledge level
feedback
Corrections and confirmed approaches
project
Ongoing work, goals, decisions
reference
Pointers to external resources

Memory File Format

---
name: user-role
description: User is a senior backend engineer
type: user
---

User is a senior backend engineer with 8+ years
of experience in Go and distributed systems.
New to the React frontend of this project.

Enable / Disable

// In settings.json
{ "autoMemoryEnabled": false }
Auto memory vs. CLAUDE.md: CLAUDE.md is written by you with explicit instructions. Auto memory is written by Claude to remember learnings. Both load at session start, but CLAUDE.md is your source of truth.

Section 11

CI/CD Workflows

Integrate Claude Code into your continuous integration and delivery pipelines. Use claude -p (print mode) to run Claude non-interactively in CI environments for automated code review, test generation, and more.

How CI/CD Works with Claude

The key to CI integration is claude -p — print mode runs Claude non-interactively, reads from stdin or arguments, and outputs to stdout. No human interaction needed.

Key: claude -p "your prompt" runs non-interactively. Combine with --model to control which model is used. All output goes to stdout for piping and capture.

Environment Variables

VariableDescriptionRequired?
ANTHROPIC_API_KEYAPI key for authenticating with ClaudeYes
CLAUDE_MODELOverride model (e.g., haiku for faster CI)No
CIAuto-set by most CI providers, signals non-interactive modeAuto-set

Claude CLI in CI

Common claude -p invocations for CI pipelines:

Command-line examples 
# Basic review
claude -p "Review the code changes in this PR"

# With model selection (faster + cheaper for CI)
claude -p --model haiku "Check for lint issues in changed files"

# Pipe diff as context
git diff origin/main...HEAD | claude -p "Review this diff for bugs and security issues"

# Output to file for CI artifacts
claude -p "Run full code review" > review-report.md

# With max turns limit for cost control
claude -p --max-turns 3 "Quick review of this PR"

# Using allowed-tools to restrict what Claude can do in CI
claude -p --allowedTools "Read,Grep,Glob" "Analyze code quality"

GitHub Actions Workflow

PR Review workflow (.github/workflows/claude-review.yml) 
name: Claude Code Review
on:
  pull_request:
    types: [opened, synchronize]

permissions:
  contents: read
  pull-requests: write

jobs:
  claude-review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Install Claude Code
        run: npm install -g @anthropic-ai/claude-code

      - name: Run Claude Review
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        run: |
          claude -p "Review the changes in this PR.
          Run git diff origin/main...HEAD and provide
          feedback on code quality, security, and
          performance." > review.md

      - name: Post Review Comment
        uses: actions/github-script@v7
        with:
          script: |
            const fs = require('fs');
            const review = fs.readFileSync('review.md', 'utf8');
            github.rest.issues.createComment({
              owner: context.repo.owner,
              repo: context.repo.repo,
              issue_number: context.issue.number,
              body: `## Claude Code Review\n\n${review}`
            });
Test Generation workflow (.github/workflows/claude-tests.yml) 
name: Claude Test Generation
on:
  pull_request:
    types: [opened, synchronize]

permissions:
  contents: write
  pull-requests: write

jobs:
  generate-tests:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
          ref: ${{ github.head_ref }}

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm

      - run: npm ci

      - name: Install Claude Code
        run: npm install -g @anthropic-ai/claude-code

      - name: Generate Missing Tests
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        run: |
          CHANGED=$(git diff --name-only origin/main -- '*.ts' '*.js' \
            | grep -v '\.test\.' | grep -v '__tests__')
          if [ -n "$CHANGED" ]; then
            claude -p "For these changed files:
            $CHANGED
            Check which ones lack tests. Generate unit
            tests for files below 80% coverage.
            Run npm test to verify they pass."
          fi

      - name: Commit Generated Tests
        run: |
          git config user.name "claude-bot"
          git config user.email "claude-bot@noreply.github.com"
          git add '*.test.*' '__tests__/**'
          git diff --staged --quiet || \
            git commit -m "test: add generated tests for changed files"
          git push

CI Best Practices

DO
  • Use claude -p for all CI commands (non-interactive)
  • Set timeout limits on CI steps (5-10 min)
  • Cache the npm install of @anthropic-ai/claude-code
  • Use --model haiku for faster, cheaper CI runs
DON'T
  • Don't use interactive mode in CI (it will hang)
  • Don't hardcode API keys in workflow files
  • Don't skip fetch-depth: 0 (Claude needs git history)
  • Don't run on every commit — use PR triggers instead
Section 12

Unit Testing with Claude

Tests are the single most impactful thing you can add to a project for Claude. They create a verification loop that lets Claude generate code, run tests, see failures, fix issues, and repeat — turning blind coding into evidence-based iteration.

Why Tests Improve Claude's Performance

Feedback Loop: Without tests, Claude operates blind — it writes code and hopes it works. With tests, Claude self-corrects: generate code, run tests, see failures, fix, repeat. This loop dramatically improves output quality.

Test-Driven Development Workflow

1. Write tests first
Ask Claude to write failing tests for the feature you want to build. This defines the expected behavior before any code exists.
2. Implement the feature
Claude writes the actual code to make the tests pass. The tests guide the implementation.
3. Run and verify
Claude runs npm test, sees results, and iterates until all tests pass. Each failure provides specific, actionable feedback.
4. Refactor with safety net
Tests ensure refactoring doesn't break anything. Claude can confidently improve code knowing tests will catch regressions.

CLAUDE.md Testing Instructions

Add these to your CLAUDE.md to enforce testing standards:

CLAUDE.md testing rules example 
## Testing Standards
- ALWAYS run `npm test` after making changes
- Write tests BEFORE implementing features (TDD)
- Minimum 80% code coverage for new code
- Test command: `npm test -- --coverage`
- Single test: `npm test -- --testPathPattern="filename"`
- Never mock what you don't own
- Every bug fix must include a regression test

## Test Structure
- Place tests in `__tests__/` mirroring src/ structure
- Name test files: `[filename].test.ts`
- Group related tests with `describe()` blocks
- Use clear test names: "should [expected behavior] when [condition]"
- Arrange-Act-Assert pattern for each test

Real Code Examples

Jest test example (TypeScript) 
// src/utils/formatPrice.ts
export function formatPrice(cents: number, currency = 'USD'): string {
  if (!Number.isFinite(cents)) throw new Error('Invalid amount');
  const dollars = (cents / 100).toFixed(2);
  return new Intl.NumberFormat('en-US', {
    style: 'currency', currency
  }).format(Number(dollars));
}

// __tests__/utils/formatPrice.test.ts
import { formatPrice } from '../../src/utils/formatPrice';

class UserService:
    def __init__(self, db):
        self.db = db

    def create_user(self, email: str, name: str) -> dict:
        if not email or '@' not in email:
            raise ValueError('Invalid email')
        if self.db.find_user(email):
            raise ValueError('Email already exists')
        return self.db.insert_user({
            'email': email.lower().strip(),
            'name': name.strip()
        })

// src/components/LoginForm.tsx
export function LoginForm({ onSubmit }: { onSubmit: (data: LoginData) => void }) {
  const [email, setEmail] = useState('');
  const [password, setPassword] = useState('');
  const [error, setError] = useState('');

  const handleSubmit = (e: FormEvent) => {
    e.preventDefault();
    if (!email) return setError('Email is required');
    if (password.length < 8) return setError('Password must be 8+ chars');
    setError('');
    onSubmit({ email, password });
  };

  return (
    <form onSubmit={handleSubmit}>
      {error && <div role="alert">{error}</div>}
      <input type="email" value={email}
        onChange={e => setEmail(e.target.value)}
        placeholder="Email" />
      <input type="password" value={password}
        onChange={e => setPassword(e.target.value)}
        placeholder="Password" />
      <button type="submit">Log In</button>
    </form>
  );
}

CLAUDE.md Git Instructions

Git workflow rules for CLAUDE.md 
## Git Workflow
- NEVER push directly to main or master
- ALWAYS create a feature branch: feature/<description>
- Commit messages: conventional commits (feat:, fix:, chore:)
- Run all tests before committing
- One logical change per commit
- Push and create PR when task is complete

## Branch Naming
- feature/<description> — new features
- fix/<description> — bug fixes
- chore/<description> — maintenance, deps, config
- refactor/<description> — code restructuring

## PR Requirements
- Title: conventional commit format (feat: add login page)
- Description: what changed and why
- Link related issues: Closes #123
- Ensure all CI checks pass before requesting review

Enforce conventional commit messages 
#!/bin/bash
# .claude/hooks/conventional-commits.sh
# Use as a PreToolUse hook on Bash matcher

INPUT=$(cat)
CMD=$(echo "$INPUT" | jq -r '.tool_input.command // empty')

# Check if this is a git commit command
if echo "$CMD" | grep -qE 'git\s+commit'; then
  # Extract the commit message
  MSG=$(echo "$CMD" | grep -oP '(?<=-m\s")[^"]+|(?<=-m\s'"'"')[^'"'"']+')

  if [ -n "$MSG" ]; then
    # Check conventional commit format
    if ! echo "$MSG" | grep -qE '^(feat|fix|chore|docs|style|refactor|test|perf|ci|build|revert)(\(.+\))?!?:\s.+'; then
      echo "BLOCKED: Commit message must follow conventional commits format." >&2
      echo "Format: type(scope): description" >&2
      echo "Types: feat, fix, chore, docs, style, refactor, test, perf, ci, build, revert" >&2
      echo "Example: feat(auth): add login endpoint" >&2
      exit 2
    fi
  fi
fi
exit 0

Git Worktree for Parallel Agents

Worktree isolation agent example 
---
name: parallel-reviewer
description: Review code in an isolated git worktree
tools: Read, Grep, Glob, Bash
model: sonnet
isolation: worktree
maxTurns: 5
---

You are a code reviewer running in an isolated worktree.
Your changes won't affect the main working directory.

1. Run `git diff origin/main...HEAD` to see all changes
2. For each changed file:
   - Read the full file for context
   - Check for bugs, security issues, and anti-patterns
   - Suggest improvements with specific line numbers
3. Run `npm test` to verify tests pass
4. Write your review to review-output.md

Because you run in a worktree, you can safely:
- Modify files to test fixes without affecting main
- Run destructive test commands
- Check out different branches for comparison

Best Practices

DO
  • Create feature branches for every task
  • Use conventional commit messages
  • Require CI checks before merge
  • Use git worktrees for parallel Claude agents
  • Keep branches short-lived (merge frequently)
DON'T
  • Push directly to main/master
  • Merge without passing status checks
  • Let Claude force-push to shared branches
  • Skip code review on Claude-generated code
  • Leave long-lived branches unmerged
Key Principle: Treat Claude-generated code the same as human-generated code. It must pass the same reviews, tests, and protection rules. Branch protection is your safety net.
